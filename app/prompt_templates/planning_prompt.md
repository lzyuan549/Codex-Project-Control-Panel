【通用规划原则】
1. 不预设技术栈、框架、语言、数据库、前端形态、部署方式或目录结构。
2. 技术方案必须基于当前项目实际情况、项目目标、上传约束文件和已有工程习惯选择。
3. 如果现有项目已经有明确技术栈、目录规范、命名风格、接口风格、测试方式或部署方式，优先沿用。
4. 如果现有项目没有明确约定，必须在 PLAN.md 中提出推荐方案、选择理由、替代方案和风险。
5. 不允许脱离当前项目空想新架构；新增结构必须服务于项目目标，并说明与现有结构的关系。
6. 功能模块、工作项和验收标准必须直接来自项目目标或现有项目需要，禁止使用无关的默认模块凑数。
7. 如题目描述不足以直接规划，先在 PLAN.md 的“待确认项”中列出问题，并给出可继续推进的保守假设。
8. 如果上传约束文件与通用原则冲突，以用户上传约束和项目实际为优先，但必须在 PLAN.md 中说明影响。

6. 不允许虚构已完成结果；规划阶段的测试记录应写“未执行 + 原因：当前仅规划阶段”。
7. 不允许出现空字段、占位符、“同上”、“待补充”等不可验收内容。

【PLAN.md 结构要求】
# Project Plan

## Goal
- 项目目标/题目原文。
- 本次目标重述：结合当前项目，说明要达成的实际结果。

## Baseline Analysis
- 当前项目结构摘要。
- 项目类型与运行方式。
- 现有技术栈与工具链。
- 现有模块、页面、接口、数据、脚本或配置摘要。
- 现有风格约定：命名、目录、错误处理、权限、状态管理、测试、部署等，按项目实际填写。
- 可复用部分。
- 本次影响范围。
- 风险与未知项。
- 待确认项。

## Recommended Approach
- 推荐实现路线。
- 技术选择依据：沿用现状、用户约束、复杂度、维护成本、风险。
- 可选替代方案及取舍。
- 与现有项目的集成方式。

## Scope
### In Scope
- 本次明确要做的内容。

### Out of Scope
- 本次不做的内容，防止范围蔓延。

## Modules / Work Areas
- 逐个列出来自项目目标的真实模块或工作域。
- 每个模块必须完整写明：来源需求原文、目标、相关用户或使用者、前置依赖、涉及入口或触发方式、核心流程、数据或状态流、接口或集成点（如适用）、实现要点、权限或安全考虑（如适用）、校验与异常、测试用例摘要、复用点与影响范围、验收标准。

## Data / API / Integration Plan
- 如果涉及数据，说明数据结构、迁移或存储方案。
- 如果涉及接口，说明接口清单、输入输出、错误处理和兼容性。
- 如果涉及外部服务、文件、任务队列、模型、浏览器、设备或第三方依赖，说明集成方式、配置项和失败处理。
- 如果不涉及本节某类内容，明确写“不涉及”并说明原因。

## Directory And File Plan
- 说明预计新增、修改、复用的目录和文件。
- 说明为什么这样组织，如何与现有项目保持一致。
- 不得强制套用固定目录；目录规划必须基于当前项目实际情况。

## Execution Checklist
- 必须展开成真实编号与真实模块名。
- 每个模块或工作域都要拆成可执行任务。
- 涉及开发时，按项目实际拆分为设计确认、数据或配置变更、核心实现、界面或交互实现、集成验证、测试验证等步骤。
- 每个执行项必须写清楚：前置依赖、具体动作、涉及文件/目录/类名/函数/接口/页面/配置、验收标准、预计产出、验证方式、风险与回滚、预计时长。
- 所有 checkbox 初始保持 `- [ ]`，不得在规划阶段写 `- [x]`。

## Test Plan
- 正常流程、边界情况、异常情况、回归范围。
- 按项目实际列出可执行验证方式，例如单元测试、接口测试、页面测试、构建检查、静态检查、人工验收等。
- 每条用例必须能记录到 TEST_REPORT.md。

## Acceptance
- 规划基于当前项目实际文件和约束。
- 模块或工作域均有清晰目标、实现要点、验收标准和测试方式。
- 执行项均具备明确动作、明确产出、明确验收、明确验证方式和预计时长。
- 风险、未知项和待确认项已列出。
- 所有测试项均可追溯到 TEST_REPORT.md。

## Notes
- 保留项目目标/题目原文，确保模块来源可追溯。
- 记录未决策事项、依赖确认事项、需用户确认事项。
- 展示 PLAN.md 预览前先自检：无空字段、无占位符、无“同上”、无不可验收任务、无过粗任务。

【TEST_REPORT.md 要求】

- 后续执行阶段每条测试记录必须包含：命令、结果（PASS/FAIL/未执行）、关键输出摘要、失败原因与处理。
